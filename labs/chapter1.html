Lab 1 — Prompt Director Warm‑Up

From coder to director: you’ll design by prompt, not by pushing pixels. This lab pairs with Chapter 1 and introduces a lightweight LLM‑first pipeline.

Open Web Mapper GPT Skip to Part A Rubric

Overview

Goal: Use natural‑language prompts to produce a clear, audience‑specific web‑map specification. You’ll practice the Prompt Director workflow (goals → constraints → critique loop), then capture your outputs in a repo you can iterate on in later labs.

What’s different: Coding and GUIs are now supporting actors. Your primary deliverable is a prompt → map‑spec → critique → refined‑spec sequence you can regenerate at will.

Learning Outcomes

  • Express a map’s purpose and audience as explicit prompt clauses.
  • Translate design constraints (accessibility, device, layers) into prompt language.
  • Run a structured critique loop and capture revisions.
  • Initialize a GitHub repo and sync with GitHub Desktop + VS Code for ongoing iterations.

Part A — Setup: GitHub + VS Code (one‑time)

  1. Create a free GitHub account.
  2. Install GitHub Desktop and VS Code.
  3. In GitHub (web), create a new repo named web‑cartography‑labs. Initialize with a README.
  4. Open GitHub Desktop → File → Clone repository → pick your repo.
  5. Open the cloned folder in VS Code. Add a folder lab‑01 and place this HTML file there (plus placeholder style.css and script.js).
  6. Commit your changes in GitHub Desktop → Publish.
  7. (Optional) Enable GitHub Pages for the repo (Settings → Pages) to host your lab files.

Part B — Prompt‑to‑Spec with Web Mapper GPT

Open Web Mapper GPT. Use the Chapter 1 Cheat Sheet scaffolds and adapt them to a dataset of your choice (or an instructor‑provided one).

Starter Prompt (copy & adapt)

You are a cartographic assistant. 
Create a map that shows [TOPIC/DATASET] for [AUDIENCE]. 
The map’s main purpose is [COMMUNICATION GOAL]. 
Constraints:
- Mobile-first, accessible to color-blind viewers (WCAG AA contrast).
- Limit to 3 legend categories and 3 interactive layers.
- Optimize labels for small screens; avoid dense text.
Return:
1) A brief rationale for audience and purpose (≤120 words).
2) A JSON-like map spec (layers, colors, symbols, interactions).
3) A self-critique and a refined alternative.
Tip: If the model returns vague or over‑engineered specs, tighten the constraints or ask it to explain trade‑offs before committing to a design.

Minimal Capture Protocol

  1. Paste your initial prompt into lab‑01/prompt.md.
  2. Export the first spec as lab‑01/spec‑v1.json (copy from the model output).
  3. Run a critique loop (ask for a self‑critique + revision). Save as spec‑v2.json.
  4. Commit your files and push to GitHub. Add short commit messages (e.g., feat: v1 spec, refine: v2 palette + legend).

Part C — Reflection (short)

In lab‑01/reflection.md, answer:

  • How did audience and purpose change the design?
  • What constraints had the biggest effect on clarity or usability?
  • What would you try differently in your next prompt?

Submission & Rubric

  • Repo initialized (GitHub + Desktop + VS Code).
  • Prompt file (prompt.md) with clear goals, audience, and constraints.
  • Two specs (spec‑v1.json, spec‑v2.json) with meaningful changes.
  • Reflection (reflection.md) that ties back to audience/purpose.
  • (Optional) GitHub Pages link to your hosted lab‑01 page.

Stretch Goals

  • Ask the model for an accessibility audit (color contrast, label sizes) and incorporate fixes.
  • Prompt the model to generate alt text for screenshots or key map states.
  • Request a bivariate alternative and compare cognitive load.

Why this Lab?

In Web Cartography 2.0, your leverage comes from language. You’re learning to encode design principles into prompts and iterate quickly.

Hook Points (for script.js)

  • Load a JSON spec preview into the page.
  • Toggle ☐/☑ on checklist items.

Common Pitfalls

  • Prompts without audience/purpose → muddy specs.
  • Too many layers → cognitive overload.
  • No critique loop → the first draft calcifies.

Copy‑Ready Prompt Variations

Design a mobile‑first choropleth of [TOPIC]. Audience: [GROUP]. Purpose: [GOAL]. Limit to 3 classes; ensure WCAG AA contrast; justify classification choice.
Propose two alternative legend designs (desktop + mobile). Critique readability and explain which better supports the audience.
Return a JSON map spec with layers, color tokens, label rules, and interactions. Then produce a refined version after a self‑critique.
Suggest an accessibility checklist for this map and flag likely failure points.
